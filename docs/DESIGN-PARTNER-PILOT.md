# Paid Design-Partner Pilot

## Recommended first commercial wedge

Sell one outcome before selling a broad platform:

> A reproducible target or biomarker evidence package assembled from public and customer-approved data, with dataset-quality assessment, harmonized metadata, workflow provenance, contradictions, limitations and review history.

Initial customers should be Indian biotechnology companies, pharmaceutical translational teams, CROs and precision-diagnostics R&D groups operating in research-use settings.

## Pilot structure

### Duration

8–12 weeks.

### Commercial hypothesis

₹5–15 lakh per pilot, excluding unusually large compute, licensed datasets, custom integrations and travel. Pricing must be tested with real buyers rather than treated as a fixed published tariff.

### Customer inputs

- one high-value scientific question;
- agreed intended use;
- named scientific and technical owners;
- approved public and private data sources;
- decision criteria;
- security questionnaire;
- weekly review availability.

### Omics Bharat deliverables

1. Project workspace and access model.
2. Evidence-search strategy.
3. Dataset-selection and exclusion log.
4. Transparent Dataset Quality Profiles.
5. Harmonized metadata dictionary.
6. One or two pinned workflow manifests.
7. Reproducible execution through the approved customer environment or configured executor.
8. Decision-ready evidence package.
9. Audit export and provenance manifest.
10. Final value and gap review.

## Success metrics

- time from question to review-ready evidence package;
- number of candidate datasets screened;
- percentage of selected datasets with complete provenance;
- hours of manual metadata cleaning avoided;
- reproducibility rate on rerun;
- number of contradictory findings explicitly surfaced;
- user satisfaction by role;
- customer willingness to renew;
- expansion opportunity to another project or team.

Do not use vanity metrics such as the number of database links or generated charts as the principal proof of value.

## Pilot acceptance criteria

A pilot is successful when:

- the customer can review every source and transformation;
- the same pinned inputs and workflow can reproduce the expected output;
- dataset-selection reasoning is visible;
- quality limitations are explicit;
- the evidence package informs a real research decision;
- the customer agrees that the result saved meaningful time or reduced risk;
- there is a defined paid next step.

## Weekly cadence

### Week 1

- intended-use workshop;
- security and data review;
- success-metric baseline;
- research-question specification.

### Weeks 2–3

- evidence discovery;
- catalog and connector gap analysis;
- dataset screening;
- initial quality profiles.

### Weeks 4–6

- metadata harmonization;
- workflow selection and pinning;
- execution and QC;
- provenance capture.

### Weeks 7–9

- cross-dataset comparison;
- contradictory evidence review;
- report generation;
- customer scientific review.

### Weeks 10–12

- revisions;
- value measurement;
- security and deployment plan;
- annual subscription proposal.

## Questions to answer before product expansion

- Which decision is painful enough to fund?
- Which dataset types recur across customers?
- Which harmonization corrections create reusable proprietary value?
- Which reports are reviewed by the same roles?
- Which integrations block adoption?
- Which controls are mandatory for procurement?
- What portion of the work remains services-heavy?
- Which parts can become repeatable product workflows?

## Stop conditions

Pause expansion if:

- every customer requires a completely different outcome;
- customers value consulting but not the workspace;
- data rights prevent commercial reuse;
- the customer cannot define a decision owner;
- the requested use is clinical or regulated without a funded validation plan;
- sensitive data cannot be placed in an approved deployment boundary.
